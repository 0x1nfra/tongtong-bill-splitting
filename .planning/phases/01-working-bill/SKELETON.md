# Walking Skeleton — Phase 1: Working Bill

> Architectural decisions for the TongTong skeleton. Subsequent phases build on these without renegotiating.

---

## What the Skeleton Delivers

The walking skeleton is a thin end-to-end slice proving the full stack works:

1. An organizer visits `/create`, enters a bill title + one item, taps "GENERATE LINK →" — a real `createBill` mutation writes to Convex and the browser redirects to `/share/[billId]`.
2. The share screen reads the real bill from Convex via `getBillForMember` and displays the title, item count, and a copyable `tongtong.my/c/[billId]` link.
3. A member opens `/c/[billId]`, sees the bill, taps "I'VE PAID" — a real `markPaid` mutation creates a `payments` record.
4. The organizer visits `/dashboard/[billId]`, sees the payment in AWAITING state, taps "STAMP SETTLED" — `confirmPayment` mutation updates the status.

After the skeleton plan executes, `pnpm dev` + `pnpm dev:convex` running concurrently produces a deployable app with real data flowing through all four routes.

## Architectural Decisions

### Framework
- **Next.js 16.2.6 App Router** — file-system routing, React Server Components for layouts, Client Components for interactive pages
- No Pages Router patterns anywhere in the codebase

### Database / Backend
- **Convex 1.39.1** — TypeScript-native mutations/queries, real-time WebSocket subscriptions via `useQuery`, file storage for DuitNow QR images
- Schema is pre-defined in `convex/schema.ts` — four tables: `bills`, `items`, `claims`, `payments`
- Convex functions live in `convex/bills.ts` and `convex/payments.ts`
- `convex/_generated/` is regenerated by running `pnpm dev:convex`

### Auth
- No login. Organizer identity = `tongtong_organizer_secret` UUID in `localStorage`
- Member identity = `tongtong_session_<billId>` UUID in `localStorage`, per-bill
- Both generated by `crypto.randomUUID()` inside `useEffect` on first visit
- Convex verifies secrets on every mutation that requires auth — no JWT, no cookies

### Deployment
- **Development:** `pnpm dev` (Next.js on :3000) + `pnpm dev:convex` (Convex local on :3210) run concurrently
- **Production:** Vercel (Next.js frontend) + Convex cloud backend — `NEXT_PUBLIC_CONVEX_URL` env var switches between local and cloud
- No Docker, no custom server — standard Vercel/Convex deployment

### State Management
- No Redux, no Zustand, no React Context beyond the Convex provider
- All server-state from `useQuery` subscriptions (auto-updating via Convex WebSocket)
- Ephemeral UI state (form fields, items being built) in component-local `useState`
- Monetary values always integer RM cents in Convex; display via `(cents / 100).toFixed(2)` with "RM" prefix

### Styling
- **Tailwind CSS v4** CSS-first config — design tokens declared in `globals.css` via `@theme inline`
- Phase 1: color tokens in use; chit CSS classes (`.chit`, `.dot-leader`, `.perforation`, `.rule-hairline`) exist but are Phase 3
- Phase 1: JetBrains Mono body font only; Departure Mono/Shadows Into Light Two/Bungee are loaded but not activated until Phase 3
